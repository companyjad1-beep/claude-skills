# Project Kickoff

새 프로젝트를 인터뷰 → 문서 → 저장소 → 서비스 연결 순서로 세팅한다.
목적: 코드를 쓰기 전에 모든 결정이 문서로 남는 것. 문서의 모든 실질 문장은 인터뷰 답변 또는 직접 조사한 사실로 추적 가능해야 한다 — 지어낸 문장이 하나라도 들어가면 이 스킬은 실패다.

## 산출물 계약 (완료 시 존재해야 하는 것)

- `CLAUDE.md` · `README.md` · `.gitignore` · `.env.example`
- `docs/PRD.md` · `docs/decisions/YYYY-MM-DD-주제.md` (결정당 파일 하나)
- UI 프로젝트만: `docs/BRAND.md` · `docs/DESIGN.md` · `docs/SCREENS.md` · `docs/UI-RULES.md` (+ 참고 브랜드 선택 시 `docs/DESIGN-REFERENCE.md`)
- git 저장소 + 첫 커밋 `chore: project kickoff`

템플릿은 [TEMPLATES.md](TEMPLATES.md). 해당 없는 섹션은 삭제한다.
`docs/UI-RULES.md`만 예외적으로 스킬 폴더의 [UI-RULES.md](UI-RULES.md)를 그대로 복사한다 — 인터뷰·수정 대상이 아니다.

## 게이트 (통과하지 못하면 다음 단계 금지)

1. **인터뷰** — [INTERVIEW.md](INTERVIEW.md) 방법론으로 진행.
   게이트: 모호성 장부 5트랙 전부 해소 + Restate 게이트(한 문장 목표) 통과.
   Refine 게이트를 통과한 결정은 그 자리에서 `docs/decisions/` 파일로 기록한다 — 세션이 끊겨도 결정은 남는다.
   **재개**: 시작 시 `docs/decisions/`가 이미 있으면 전부 읽고 장부를 복원한 뒤, 해소된 트랙은 다시 묻지 않고 남은 트랙부터 이어간다.
2. **문서 생성** — 게이트: 각 문서의 내용이 인터뷰 답변·조사 사실로 추적됨. 안 나온 내용은 비우거나 섹션째 삭제.
   참고 브랜드가 정해지면(인터뷰 중이어도 즉시) `npx -y getdesign@latest add <브랜드>`로 해당 디자인 시스템을 받아 `docs/DESIGN-REFERENCE.md`로 저장한다(getdesign.md 카탈로그에 없으면 건너뜀). BRAND/DESIGN의 컬러·타이포 값은 이 파일에서 추려 사용자 확인 후 채운다 — 레퍼런스는 재료일 뿐, 결정 문서는 BRAND/DESIGN이다.
3. **저장소 초기화** — git init + 스택 표준 .gitignore + 첫 커밋.
   게이트: `.env*`가 ignore되고(`.env.example`만 예외) 커밋 내용에 시크릿이 없음을 확인.
4. **외부 서비스 연결** — 인터뷰 트랙 ③에서 정한 서비스만. 로그인·과금이 걸린 명령은 사용자가 직접 실행하게 안내(예: `! supabase login`, `! vercel login`). 명령마다 "지금 뭘 하는 건지" 한 줄 설명. 발급된 키는 `.env`에만 넣게 안내.
   게이트: 서비스별로 연결 완료 또는 (사용자가 "나중에"라면) README에 TODO 기록.
5. **최종 검증** — 실제 생성된 파일 목록과 문서의 핵심 결정 요약을 장부와 대조해 보여주고 사용자 확인을 받는다. 파일을 직접 확인하지 않은 채 완료 선언 금지. 수정 요청은 반영 후 커밋.

## 경계

- 디렉토리가 비어있지 않으면 파일을 하나라도 만들기 전에 알리고 진행 여부를 확인받는다. 예외: 중단된 킥오프의 `docs/decisions/`만 있는 경우 — 확인 없이 재개한다.
- 스캐폴딩(create-next-app 등)은 범위 밖. 세팅 완료 후 "이제 스캐폴딩 할까요?"로 제안만 한다.
- 멈추는 경우는 셋뿐 — 파괴적 작업, 범위 변경, 사용자만 답할 수 있는 판단. 그 외에는 묻지 않고 진행한다.
- 문서는 전부 한국어(코드 식별자·기술 용어는 영어 그대로).
