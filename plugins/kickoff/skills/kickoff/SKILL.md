# Project Kickoff

새 프로젝트의 기반을 인터뷰 → 문서 → 검증 순서로 세팅한다.
목표: 나중에 "이거 왜 이렇게 했지?"가 안 나오게, 결정을 전부 문서에 남기고 시작.

## 진행 순서 (체크리스트로 관리)

1. **사전 확인**: 현재 디렉토리가 비어있는지 확인. 기존 파일이 있으면 사용자에게 알리고 진행 여부 확인.
2. **딥 인터뷰**: [INTERVIEW.md](INTERVIEW.md)의 소크라테스식 방법론으로 진행.
   - 모호성 장부(5개 트랙)를 만들고, 가장 큰 모호성을 겨냥해 한 번에 질문 하나씩.
   - 질문마다 추천안 제시. 사실은 직접 조사하고, 판단만 사용자에게 묻는다.
   - 긴 답변은 Refine 게이트로 구조화, 종료 전 Restate 게이트(한 문장 목표) 통과 필수.
3. **문서 생성**: [TEMPLATES.md](TEMPLATES.md) 템플릿에 인터뷰 답변을 채워 생성:
   - `CLAUDE.md` — 프로젝트 규칙 (스택, 코딩 규칙, 금지사항, 문서 지도)
   - `docs/PRD.md` — 무엇을 왜 만드는지, MVP 범위, 제외 범위
   - `docs/BRAND.md` — 브랜드 정체성, 컬러 팔레트, 톤앤매너, 보이스
   - `docs/DESIGN.md` — 디자인 컨셉, 타이포, 간격·형태, 컴포넌트 규칙
   - `docs/SCREENS.md` — 화면 목록, 하단 네비 탭 구성, 이동 흐름
   - `docs/UI-RULES.md` — 스킬 폴더의 [UI-RULES.md](UI-RULES.md)를 그대로 복사 (하단 네비 겹침, safe area, z-index 등 공통 버그 예방 고정 규칙 — 인터뷰 불필요)
   - `docs/decisions/` — 결정 하나당 파일 하나(`YYYY-MM-DD-주제.md`). 인터뷰 결정들을 각각 파일로 생성, 이후 개발하며 계속 추가
   - `README.md` — 프로젝트 한 줄 소개 + 실행 방법 자리
   - `.env.example` — 인터뷰에서 나온 외부 서비스들의 환경변수 키 이름만 미리 생성 (값 비움)
4. **저장소 초기화**: `git init` + 스택에 맞는 `.gitignore` 생성 + 첫 커밋 (`chore: project kickoff`).
5. **외부 서비스 연결**: 인터뷰 트랙 ③에서 정한 서비스(Supabase, Vercel, Firebase 등)를 초보자 기준 단계별로 연결한다.
   - CLI 설치 확인 → 없으면 설치 명령 안내 → 로그인은 사용자가 `! <명령>`으로 직접 실행하게 안내 (예: `! supabase login`, `! vercel login`)
   - 프로젝트 생성/연결(link)까지 진행하고, 발급된 키를 `.env`에 채우는 것 안내 (`.env`는 커밋 금지 재확인)
   - 각 단계마다 "지금 뭘 하는 건지" 한 줄 설명 붙이기. 사용자가 "나중에"라고 하면 건너뛰고 README에 TODO로 남긴다.
6. **최종 검증**: 생성된 문서를 요약해 보여주고, 인터뷰 답변과 어긋난 부분이 없는지 사용자 확인을 받는다. 수정 요청이 있으면 반영 후 커밋.

## 규칙

- 템플릿의 모든 섹션을 억지로 채우지 않는다. 해당 없으면 섹션 삭제.
- 인터뷰에서 안 나온 내용을 지어내서 문서에 넣지 않는다.
- BRAND.md / DESIGN.md / UI-RULES.md는 UI 있는 프로젝트일 때만 생성.
- UI-RULES.md는 인터뷰 없이 그대로 복사. 프로젝트별로 임의 수정하지 않는다.
- 스캐폴딩(create-next-app 등)은 이 스킬 범위 밖. 문서 세팅 완료 후 "이제 스캐폴딩 할까요?"로 넘긴다.
- 문서는 전부 한국어로 작성 (코드 식별자·기술 용어는 영어 그대로).
