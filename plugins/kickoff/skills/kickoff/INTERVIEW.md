# 킥오프 딥 인터뷰

목적: 문서를 쓸 수 있을 만큼만 모호성을 없앤다. 인터뷰 자체가 목적이 아니다 — 장부가 다 해소되면 즉시 멈춘다.

## 원칙

- **한 번에 질문 하나.** 가장 큰 모호성을 겨냥하고, 질문마다 추천안을 함께 제시한다(첫 옵션에 "(추천)"). 사용자가 "모르겠다"로 도망칠 수 없게.
- **사실은 조사, 판단은 사용자.** 라이브러리 기능·가격·API 제약은 직접 검색해 "~로 확인했는데 맞나요?"로 통보만. 목표·범위·취향·트레이드오프는 사용자에게. 의심스러우면 사용자에게 — 인터뷰 상대는 코드가 아니라 사람이다.
- **결정된 것은 다시 묻지 않는다.** 대신 이전 답과 모순이 생기면(예: 출시 2주 + 기능 10개) 그 자리에서 지적하고 조정한다.
- **초보자 눈높이.** 역질문("A랑 B 뭐가 나아?", "그게 뭐야?")에는 옵션을 다시 내밀지 말고 개념과 차이를 쉬운 말로 설명(필요하면 비교표)한 뒤 추천과 함께 답한다. 전문 용어는 첫 등장에 한 줄 풀이(예: "Edge Function = 서버 없이 돌아가는 작은 서버 코드"). 결정에 영향 주는 배경지식은 묻기 전에 먼저 알려준다.
- **존재론적 질문 활용.** "이건 본질적으로 무엇인가?", "그건 원인인가 증상인가?", "지금 우리가 가정하고 있는 건 뭔가?"

## 모호성 장부 — 해소 조건이 곧 게이트

| 트랙 | 해소 조건 |
|------|----------|
| ① 제품 정의 | 무엇을·누구를 위해·왜, 성공 기준이 한 문장씩 나옴 |
| ② 범위 | MVP 기능 목록 + **명시적 제외 목록**(스코프 크립 방지) + 목표 시점 |
| ③ 기술 스택 | 플랫폼, 프레임워크, DB/인증/결제 등 외부 서비스, 이미 보유한 계정 |
| ④ 브랜드/디자인 | 브랜드 한 줄, 톤앤매너 형용사 3개, 참고 브랜드, 컬러 팔레트, 보이스(말투), 디자인 컨셉, 다크모드, 모바일/데스크톱 우선, 핵심 화면 목록 + 하단 네비 탭 구성 (UI 없으면 트랙 삭제) |
| ⑤ 운영 규칙 | 테스트 정책, 배포처, Claude 금지사항, 커밋 규칙 |

한 트랙만 파고들지 말 것. 몇 라운드마다 장부 현황을 보여주며 남은 트랙을 환기한다.
트랙 ④ 지름길: 참고 브랜드가 [getdesign.md](https://getdesign.md) 카탈로그(Vercel·Airbnb·Claude 등 70여 개)에 있으면 "그 브랜드 디자인 시스템을 시작점으로 가져올까요?"를 먼저 제안한다. 수락하면 그 즉시 받아두고(SKILL.md 게이트 2 방식), 컬러 팔레트·타이포는 처음부터 묻지 않고 가져온 값에서 바꿀 것만 확인받는다.

## Refine 게이트 (긴 답변 구조화)

이유·제약·범위가 담긴 긴 답은 한 줄로 압축하지 말고 아래 구조로 정리해 되돌려주고 빠진 것이 없는지 확인받는다:

```
결정: <핵심 결정>
이유: <사용자가 말한 이유>
제약: <명시된 제약>
제외 범위: <명시적으로 뺀 것>
```

예/아니오·고유명사 하나짜리 답은 건너뛴다. 확인받은 즉시 `docs/decisions/YYYY-MM-DD-주제.md` 파일로 기록한다 — 문서 생성 단계를 기다리지 않는다. 이 파일들이 인터뷰의 진행 상태이며, 세션이 끊겨도 여기서 재개한다.

## 종료 — Restate 게이트

5트랙이 해소되거나 사용자가 "이 정도면 됐다"고 하면 종료 수순으로 간다. 문구 다듬기나 좁은 엣지케이스만 남았으면 더 파지 않는다.

종료 전, 합의 내용을 **한 문장 목표**로 압축해 확인받는다:

> "합의한 내용을 한 문장으로 정리하면: `<한 문장>` — 이 문장만 읽은 다른 사람도 같은 결과물을 떠올릴까요?"

옵션: ① 맞다, 문서 생성 시작 ② 문구 수정 ③ 빠진 범위 있음.
②·③이면 장부에 반영하고 Restate를 한 번만 재실행한다. 두 번 넘게 돌지 않는다 — 그래도 안 맞으면 정체된 트랙에 대한 직접 질문으로 돌아간다.

통과하면 SKILL.md 게이트 2(문서 생성)로 진행. 한 문장 목표는 PRD 맨 위에 그대로 들어간다.
