# 문서 템플릿

인터뷰 답변으로 `<>` 자리를 채운다. 해당 없는 섹션은 삭제 — 억지로 채우지 않는다.
인터뷰에서 안 나온 내용을 지어내지 않는다.

## CLAUDE.md

```markdown
# <프로젝트명>

<한 문장 목표 — Restate 게이트에서 확정된 문장>

## 스택
- <플랫폼/프레임워크/DB/인증 등, 버전 포함>

## 규칙
- <테스트 정책>
- <커밋/브랜치 규칙>
- <금지사항: 사용자가 Claude에게 하지 말라고 한 것>

## 문서 지도
- 기획: docs/PRD.md
- 브랜드(컬러·톤앤매너·보이스): docs/BRAND.md
- 디자인 규칙(컨셉·타이포·컴포넌트): docs/DESIGN.md
- 화면 구조(화면 목록·네비 탭·이동 흐름): docs/SCREENS.md — **새 화면은 여기 먼저 추가**
- UI 공통 규칙(겹침·safe area·z-index — 수정 금지): docs/UI-RULES.md
- 결정 기록: docs/decisions/ — **중요한 결정마다 `YYYY-MM-DD-주제.md` 파일 하나 생성** (결정/이유/제외한 대안). 관련 작업 전에 해당 주제 파일 먼저 확인

## UI 작업 규칙
- 화면을 만들거나 수정하기 전에 반드시 docs/BRAND.md, docs/DESIGN.md, docs/UI-RULES.md를 읽고 준수한다.
- 색상·크기·간격은 문서에 정의된 토큰/스케일만 사용. 임의 값 금지.
- 새 컴포넌트를 만들기 전에 components/에 비슷한 게 이미 있는지 먼저 확인. 중복 생성 금지.

## 코드 스타일 (단순하게)
- 동작하는 가장 단순한 코드가 정답. 화려한 것보다 지루한 코드.
- 순서대로 검토 후 첫 번째로 되는 걸로: ① 애초에 필요한가? (나중에 필요할 것 같은 건 안 만듦) ② 언어/프레임워크 기본 기능으로 되나? ③ 이미 설치된 라이브러리로 되나? ④ 그래도 안 되면 최소한의 코드.
- 구현이 하나뿐인 인터페이스, 나중을 위한 추상화·설정·스캐폴딩 금지.
- 파일은 최소로, diff는 짧게. 추가보다 삭제가 좋은 코드.

## 작업 규칙 (상습 실수 방지)
- PRD의 제외 범위에 있는 기능은 요청받아도 만들지 않고 먼저 확인받는다. 요청 안 된 기능을 임의로 추가하지 않는다.
- 새 라이브러리 설치 전에 기존 것/직접 구현 몇 줄로 되는지 먼저 확인. 설치하면 docs/decisions/에 이유 기록.
- 에러·경고를 무시하고 다음 작업으로 넘어가지 않는다. 빌드가 깨진 채로 새 기능 시작 금지.
- 기능 하나 완성될 때마다 커밋. 여러 기능을 한 커밋에 뭉치지 않는다.
- 완성 선언 전에 실제로 실행/빌드해서 확인. 확인 없이 "됐습니다" 금지.

## 폴더 구조 (여기 정의된 곳 외에 파일 생성 금지)
<스택에 맞는 구조 — 예: Next.js면>
- `app/` — 라우트만
- `components/` — 재사용 컴포넌트 (`components/ui/`는 기본 요소)
- `lib/` — 유틸, API 클라이언트, 상수
- `docs/` — 문서

## 시크릿 규칙
- API 키·비밀번호는 `.env`에만. 코드에 하드코딩 절대 금지.
- 새 환경변수 추가 시 `.env.example`에 키 이름만(값 없이) 함께 기록.
```

```markdown
## 문제와 사용자

- 문제: <이게 없으면 지금 어떻게 하고 있는가>
- 성공 기준: <측정 가능한 기준>

## MVP 범위
1. <기능 1>
2. <기능 2>
3. <기능 3>

## 제외 범위 (일부러 안 만드는 것)
- <기능> — <뺀 이유>

## 일정·수익

- 유료화: <무엇에 돈을 받는가 / 없음>

## 마일스톤 (작업 순서)
1. <MVP 기능 중 가장 먼저 — 세로로 관통하는 한 조각>

3. <다음>
```

```markdown
- 참고 브랜드: <서비스명 — 어떤 점을 참고하는지>

## 컬러
- Primary: <#hex> — <쓰임: CTA, 강조>
- Secondary: <#hex>
- Background: <#hex> / Surface: <#hex>
- Text: <#hex> / Text-muted: <#hex>
- Success/Warning/Error: <#hex 3개>
- 다크모드: <지원 여부 — 지원 시 다크 팔레트도 정의>

## 보이스 (문구 작성 규칙)
- 말투: <해요체/합니다체/반말>

- 예시: 버튼은 "<시작하기>"처럼 <동사형/명사형>으로
```

## docs/DESIGN.md (UI 있는 프로젝트만)

```markdown
# 디자인 규칙

## 컨셉
- 디자인 방향: <미니멀/브루탈리즘/글래스모피즘/...>
- 우선 기기: <모바일/데스크톱> · 언어: <한국어/...>

## 타이포그래피
- 제목: <폰트/굵기> / 본문: <폰트> (한국어면 Pretendard 기본 추천)

## 간격·형태
- 간격 단위: <4px 배수만 사용>
- 모서리: <radius 값 — 예: 카드 12px, 버튼 8px, 이 외 금지>
- 그림자: <단계 정의 또는 사용 안 함>

## 컴포넌트 규칙
- <버튼/카드/입력창 등 인터뷰에서 나온 규칙만>

## 공통 UI 규칙
- docs/UI-RULES.md 준수 (하단 네비 겹침, safe area, z-index 등 — 수정 금지)
```

## docs/SCREENS.md (UI 있는 프로젝트만)

```markdown
# 화면 구조

## 하단 네비 탭 (앱형 UI일 때)
| 탭 | 화면 | 아이콘 |
|-----|------|--------|
| <홈> | <화면명> | <아이콘> |
<탭은 3~5개. 5개 초과 금지.>

## 화면 목록
| 화면 | 경로 | 진입점 | 핵심 역할 한 줄 |
|------|------|--------|----------------|
| <홈> | </> | <네비 탭> | <...> |
| <상세> | </items/[id]> | <홈에서 카드 클릭> | <...> |

## 이동 흐름
<핵심 플로우만: 예> 홈 → 상세 → 결제 → 완료(→ 홈으로 복귀)

## 규칙
- 새 화면 추가 시 이 표에 먼저 추가하고 만든다.
- 모든 화면은 UI-RULES.md의 상태 4종(로딩/빈/에러/정상)을 갖춘다.
- 뒤로가기 시 어디로 가는지 모호한 화면을 만들지 않는다.
```

## docs/UI-RULES.md (UI 있는 프로젝트만)

스킬 폴더의 `UI-RULES.md`를 **그대로 복사**한다. 인터뷰로 정하는 내용이 아님 — 모든 프로젝트 공통 고정 규칙.

## docs/decisions/ (결정 하나 = 파일 하나)

파일명이 곧 색인이다: `YYYY-MM-DD-주제.md` — 파일명만 보고 내용을 알 수 있게 짓는다.
킥오프 시 인터뷰의 Refine 게이트 통과 답변을 **결정별로 각각 파일로** 만든다
(예: `2026-07-02-스택-nextjs-supabase.md`, `2026-07-02-범위-mvp-3기능-확정.md`).

각 파일 형식:

```markdown
# <결정 한 줄>

- 날짜: <YYYY-MM-DD>
- 결정: <무엇을 하기로 했나>
- 이유: <왜>
- 제약: <명시된 제약>
- 제외한 대안: <뭘 안 하기로 했고 왜>
```

## .gitignore

스택에 맞는 표준 gitignore 사용 (Node면 node_modules/.env/.next 등). `.env*`는 스택 불문 항상 포함 (`.env.example`만 예외로 커밋).

## .env.example

외부 서비스 인터뷰 답변(트랙 ③)에서 나온 서비스들의 키 이름만 미리 채운다 (값은 비움):

```
# 예: Supabase 쓴다고 답했으면
NEXT_PUBLIC_SUPABASE_URL=
SUPABASE_SERVICE_ROLE_KEY=
```
